What is RAG?
Retrieval-Augmented Generation (RAG) is an architecture that makes LLMs smarter about specific information they weren't trained on — like your own documents, a company's knowledge base, or real-time data.

The core problem RAG solves: LLMs have a knowledge cutoff and no awareness of your private data. RAG fixes this by retrieving relevant context at query time and injecting it into the prompt before the LLM answers.

Why not just fine-tune? Fine-tuning is expensive, slow, and you'd have to redo it every time data changes. RAG is dynamic — update the knowledge base and the chatbot immediately knows more.

Key Concepts to Internalize
Embeddings are numerical representations (vectors) of text. The magic is that semantically similar text has similar vectors — so "how do I reset my password" and "forgot my login credentials" would have very close vectors, even though the words differ. An embedding model converts text into these vectors (e.g. 768 or 1536 numbers).

Vector database is a special database optimized for similarity search — given a query vector, find the N closest stored vectors. Popular options: ChromaDB (local, free), Qdrant, Pinecone. For a beginner project, ChromaDB is perfect because it runs entirely in-memory locally.

Chunking strategy matters more than people expect. If chunks are too large, you retrieve irrelevant noise. Too small, and you lose context. ~300–500 tokens with 50-token overlap is a solid default.

The prompt construction is the glue. After retrieval, you build a prompt like:

You are a helpful assistant. Answer the question using ONLY the context below.
Context: [retrieved chunk 1] [retrieved chunk 2] ...
Question: [user's question]

Step 3 — Full RAG Chatbot Code
Here's a minimal but complete RAG chatbot. Paste this into Colab or a .py file:

python
import os
from groq import Groq
import chromadb
from sentence_transformers import SentenceTransformer

# ── Config ──────────────────────────────────────────────
GROQ_API_KEY = "your_groq_api_key_here"
LLM_MODEL    = "llama3-8b-8192"   # free on Groq
EMBED_MODEL  = "all-MiniLM-L6-v2" # runs locally, no API needed
TOP_K        = 3                   # number of chunks to retrieve

# ── Clients ─────────────────────────────────────────────
groq_client    = Groq(api_key=GROQ_API_KEY)
embed_model    = SentenceTransformer(EMBED_MODEL)
chroma_client  = chromadb.Client()
collection     = chroma_client.get_or_create_collection("knowledge_base")

# ── Indexing ─────────────────────────────────────────────
def add_documents(texts: list[str]):
    """Chunk, embed and store a list of text strings."""
    embeddings = embed_model.encode(texts).tolist()
    ids        = [f"doc_{i}" for i in range(len(texts))]
    collection.add(documents=texts, embeddings=embeddings, ids=ids)
    print(f"Indexed {len(texts)} chunks.")

# ── Retrieval ────────────────────────────────────────────
def retrieve(query: str) -> list[str]:
    q_vec   = embed_model.encode([query]).tolist()
    results = collection.query(query_embeddings=q_vec, n_results=TOP_K)
    return results["documents"][0]  # list of top-k chunks

# ── Generation ───────────────────────────────────────────
def ask(query: str) -> str:
    chunks  = retrieve(query)
    context = "\n\n".join(chunks)
    prompt  = (
        f"Answer the question using ONLY the context below. "
        f"If the answer isn't in the context, say 'I don't know'.\n\n"
        f"Context:\n{context}\n\n"
        f"Question: {query}"
    )
    response = groq_client.chat.completions.create(
        model=LLM_MODEL,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content

# ── Example usage ────────────────────────────────────────
if __name__ == "__main__":
    # Add your knowledge base here
    docs = [
        "ChromaDB is a vector database that runs locally in Python.",
        "Groq provides fast LLM inference using custom LPU hardware.",
        "RAG stands for Retrieval-Augmented Generation.",
        "Embeddings are numerical representations of text that capture semantic meaning.",
        "The sentence-transformers library provides free local embedding models.",
    ]
    add_documents(docs)

    # Chat loop
    print("RAG Chatbot ready. Type 'quit' to exit.\n")
    while True:
        q = input("You: ")
        if q.lower() == "quit":
            break
        print(f"Bot: {ask(q)}\n")

import pypdf

def load_pdf(path: str, chunk_size: int = 500) -> list[str]:
    reader = pypdf.PdfReader(path)
    full_text = " ".join(page.extract_text() for page in reader.pages)
    # Simple fixed-size chunking with overlap
    words  = full_text.split()
    chunks = []
    step   = chunk_size - 50  # 50-word overlap
    for i in range(0, len(words), step):
        chunks.append(" ".join(words[i:i + chunk_size]))
    return chunks

Phase 2 — The 4 building blocks of RAG
Before writing a single line of RAG code, you need to understand what's actually happening under the hood. Each building block is simple on its own. Let's go one at a time.

Building Block 1 — Embeddings
This is the most important concept in all of RAG. Everything else depends on understanding this.

The core problem: Computers can't understand meaning. They only understand numbers. So how do you search for meaning in text rather than just matching keywords?

For example — imagine your PDF contains this sentence:

"The patient should avoid consuming sodium."

And the user asks:

"Can I eat salty food?"

A normal keyword search would find zero matches — none of the words overlap. But a human immediately knows these are about the same thing.

Embeddings solve this. An embedding model reads a piece of text and converts it into a list of numbers — typically 300 to 1500 numbers long. These numbers aren't random — they're carefully trained so that text with similar meaning produces similar numbers.

Visually, imagine every sentence as a point floating in space. "Avoid sodium" and "can I eat salty food" end up as points very close together. "How do I fix a car engine" ends up far away from both.

This is called a vector — just a list of numbers. And the "space" they float in is called vector space.

Here's the key insight:

Similarity in meaning = closeness in vector space

That's the entire foundation of RAG retrieval.

A concrete example of what an embedding looks like:

The sentence "The cat sat on the mat" might become something like:

[0.23, -0.87, 0.14, 0.56, -0.32, 0.91, ... ] # 768 numbers total
The actual numbers mean nothing to you or me. But the embedding model trained on billions of sentences has encoded semantic meaning into those positions.

Two things to remember:

The same embedding model must be used for both indexing and querying — otherwise the numbers are on different scales and comparison breaks
Embedding is done by a separate model from the LLM. The LLM generates text. The embedding model just converts text to vectors.

Building Block 2 — Vector Search
Now you have a database full of vectors (one per chunk of your document). The user asks a question. You embed the question. Now what?

You need to find which stored vectors are closest to the query vector. This is called similarity search.

The most common way to measure closeness is called cosine similarity. Don't worry about the math — the intuition is what matters.

Imagine two arrows pointing out from the center of a circle. Cosine similarity measures the angle between them:

Angle of 0° → identical meaning → similarity score of 1.0
Angle of 90° → unrelated → similarity score of 0.0
Angle of 180° → opposite meaning → similarity score of -1.0
So when you search, the vector database computes this angle between your query vector and every stored vector, then returns the top K closest ones. That's literally it.

Top-K just means "give me the K most similar chunks." If K=3, you get back the 3 chunks from your document most relevant to the user's question.

Why not just use a regular database?

A regular database like SQL is great for exact matches — "find all rows where name = 'John'". But it has no concept of closeness. You can't ask SQL "find me rows that are kind of similar to this." Vector databases are built specifically for that.

For your project you'll use ChromaDB — it runs entirely on your machine, no account needed, and is designed exactly for beginners building RAG systems.

Putting blocks 1 and 2 together so far:

Your PDF
   ↓ split into chunks
   ↓ each chunk → embedding model → vector
   ↓ store (chunk text + its vector) in ChromaDB

User question
   ↓ embedding model → vector
   ↓ compare against all stored vectors (cosine similarity)
   ↓ return top 3 most similar chunks

Building Block 3 — Prompt Construction
You now have:

The user's question
The top K chunks retrieved from your document
Now you need to feed both to the LLM in a way that tells it exactly what to do. This is called prompt construction — you're manually building the message that gets sent to the LLM.

Here's what a basic RAG prompt looks like:

You are a helpful assistant. Answer the user's question using ONLY 
the context provided below. If the answer is not in the context, 
say "I don't know based on the provided document."

Context:
[chunk 1 text goes here]

[chunk 2 text goes here]

[chunk 3 text goes here]

Question: [user's question goes here]
That's it. You're just doing string formatting — slotting the retrieved chunks and the question into a template.

Why does "ONLY the context" matter?

Without that instruction, the LLM will answer from its general training data — which defeats the entire purpose of RAG. You want it grounded to your document. That phrase is what keeps it honest.

This is also why the instruction "if the answer is not in the context, say I don't know" is important. Otherwise the LLM will confidently make something up — this is called hallucination, and it's the biggest practical problem with LLMs.

A concrete example:

Say your PDF is a company HR policy document. A chunk says:

"Employees are entitled to 18 days of paid leave per year."

User asks: "How many days of leave do I get?"

Your constructed prompt becomes:

You are a helpful assistant. Answer using ONLY the context below.

Context:
Employees are entitled to 18 days of paid leave per year.

Question: How many days of leave do I get?
The LLM reads this and answers: "According to the document, you are entitled to 18 days of paid leave per year."

No hallucination. Grounded. Traceable back to your document.

One more thing — system vs user messages

When you call an LLM API, you don't just send one blob of text. You send a structured list of messages, each with a role:

system — background instructions that set the LLM's behaviour. The LLM treats this as its operating rules.
user — the actual human message
assistant — the LLM's previous replies (used when you have a multi-turn conversation)
In code this looks like:

python
messages = [
    {"role": "system", "content": "You are a helpful assistant. Answer using only the provided context."},
    {"role": "user", "content": "Context:\n[chunks here]\n\nQuestion: How many days of leave do I get?"}
]
You'll use this exact structure when calling the Groq API.

Building Block 4 — The LLM API Call
Everything so far has been local — chunking, embedding, storing, retrieving. But the final step (generating the actual answer) is done by an LLM running on someone else's server. You communicate with it over the internet by calling an API.

What is an API?

Think of it like a waiter at a restaurant. You (your code) don't go into the kitchen yourself. You give your order to the waiter (the API), the kitchen (the LLM server) prepares it, and the waiter brings the result back to you.

In code, you're just sending an HTTP request — your prompt goes out, the answer comes back. That's the entire interaction.

What Groq is

Groq is a company that hosts powerful open-source LLMs (like Llama 3, Mixtral) on their servers and lets you call them for free via API. Their hardware is unusually fast — responses come back in under a second typically.

Their API follows the exact same structure as OpenAI's API — so if you ever learn one, you know the other.

What an API key is

When you sign up on Groq's website, they give you an API key — a long string like gsk_abc123xyz.... This is your identity. Every request you make includes this key so Groq knows who's calling and can track your usage against the free limit.

You never hardcode this key directly in your code files — because if you accidentally share your code, someone else can use your key and exhaust your credits. Instead you store it in a separate file called .env and read it from there. We'll cover this properly when we build.

Good that you're asking this before running the code. Let me explain it properly.

What is chunking and why do we do it?
Remember in Phase 2 we talked about embeddings — you convert text into a vector so you can search by meaning. Here's the problem: you can't embed an entire PDF as one vector.

Imagine your PDF is a 50-page textbook. If you embed the whole thing as one vector, that vector is trying to represent everything — every topic, every paragraph. When the user asks a specific question, you'd retrieve the entire 50 pages and dump it into the LLM's prompt. That's way too much text, and the LLM would get confused or hit its input limit.

So instead you cut the document into small pieces — chunks. Each chunk gets its own vector. Now when the user asks something, you retrieve only the 2-3 chunks actually relevant to that question.

Think of it like an index at the back of a textbook. Instead of re-reading the whole book to find something, you look up the specific pages. Chunks are those specific pages.

What is chunk_size?
chunk_size=200 means each chunk is 200 words long.

Imagine your document has this text (simplified):

The company was founded in 1998. It started as a small 
garage startup. By 2005 it had grown to 500 employees. 
The CEO at the time was John Smith. He led the company 
through a major expansion into Asian markets...
With chunk_size=200, you'd take the first 200 words as chunk 1, then the next 200 words as chunk 2, and so on until the document ends.

What is overlap and why does it exist?
This is the part most people don't immediately get. Let me show you the problem it solves.

Say you have this sentence right at the boundary between chunk 1 and chunk 2:

...end of chunk 1: "The merger was announced in"
start of chunk 2: "December 2003 after months of..."
The sentence "The merger was announced in December 2003" got split across two chunks. Neither chunk has the complete thought. If the user asks "when was the merger announced?", neither chunk alone gives a good answer.

Overlap fixes this by making consecutive chunks share some words. The end of chunk 1 also appears at the start of chunk 2. So the full sentence stays intact in at least one chunk.

Visually:

Words:    1 -------- 50 -------- 100 -------- 150 -------- 200 ...

Chunk 1:  |___________________________|
                                 (overlap)
Chunk 2:                    |___________________________|
                                              (overlap)
Chunk 3:                               |___________________________|
Each chunk shares 50 words with the next one. No sentence gets lost at the boundary.

What is step?
step is simply how far you jump forward before starting the next chunk.

Think of it with a physical example. Imagine you have a 10-metre rope and you're cutting it into 3-metre pieces, but you want each piece to share 1 metre with the next:

Piece 1 starts at 0m, ends at 3m
Piece 2 starts at 2m (not 3m), ends at 5m — shares 1 metre with piece 1
Piece 3 starts at 4m, ends at 7m — shares 1 metre with piece 2
The jump between starts is 2m. That jump is the step — chunk_size - overlap = 3 - 1 = 2.

In your code: step = 200 - 50 = 150. So:

Chunk 1: words 0   to 200
Chunk 2: words 150 to 350   ← jumped 150 forward, shares words 150-200 with chunk 1
Chunk 3: words 300 to 500   ← jumped 150 forward, shares words 300-350 with chunk 2
How does storing fit in?
After chunking you have something like this in memory:

python
chunks = [
    "The company was founded in 1998...",   # chunk 1
    "...grew to 500 employees. The CEO...", # chunk 2
    "...expansion into Asian markets...",   # chunk 3
    # ... and so on
]
These are just strings sitting in your program's memory. The moment your program stops running, they're gone.

So you embed each chunk (convert to a vector) and store both the text and its vector in ChromaDB. ChromaDB saves this to disk so it persists. Later when the user asks a question, ChromaDB is still there with all your chunks ready to be searched.

Great question — this is the most intellectually interesting part of all of RAG. Let me build it up from scratch.

Step 1 — The fundamental problem
Computers only understand numbers. The word "cat" means nothing to a computer. So the first challenge ever in NLP (Natural Language Processing) was: how do we turn words into numbers in a meaningful way?

The naive approach was simple — just assign every word a number. "cat" = 1, "dog" = 2, "car" = 3. But this is useless because the numbers have no relationship to meaning. Why would "cat" = 1 and "dog" = 2 imply they're related? They're just arbitrary IDs.

Step 2 — The first real idea: meaning comes from context
In the 1950s, linguist John Firth said something that became the foundation of all modern NLP:

"You shall know a word by the company it keeps."

Think about what this means. The word "bank" appears in sentences like:

"I deposited money at the bank"
"The bank approved my loan"
"I sat by the river bank"
"The bank was muddy after the rain"
The first two sentences have "bank" surrounded by words like "money", "loan", "deposit". The last two have "bank" surrounded by "river", "muddy", "water". Same word, two completely different meanings — and the surrounding words tell you which meaning is being used.

This insight led to the idea: train a model on massive amounts of text, and let it learn word meaning purely from context.

Step 3 — Word2Vec (the breakthrough, 2013)
Google researchers built a neural network called Word2Vec that did exactly this. They fed it billions of sentences and trained it with a simple task:

Given the surrounding words, predict the missing word.

For example:

"The cat sat on the ___"
The model has to predict "mat". It does this millions of times across billions of sentences, adjusting its internal numbers every time it gets it wrong.

After training, something remarkable emerged. Each word got represented as a vector — a list of numbers. And these numbers captured semantic relationships:

vector("king") - vector("man") + vector("woman") ≈ vector("queen")
The model learned that king and queen have the same relationship as man and woman — purely from reading text. Nobody told it that. It figured it out from context patterns.

This is the core magic. Meaning is encoded as geometry in vector space.

Step 4 — From words to sentences to chunks
Word2Vec gave vectors to individual words. But you need vectors for entire chunks of text — paragraphs, sentences, passages. How do you go from word vectors to chunk vectors?

This is where transformer models come in — the architecture behind BERT, GPT, and every modern LLM.

Transformers introduced a mechanism called self-attention. Here's the intuition:

When processing the sentence:

"The animal didn't cross the street because it was too tired"
What does "it" refer to — the animal or the street? Humans instantly know it's the animal. Self-attention lets the model figure this out too by looking at every word's relationship to every other word simultaneously.

For each word, the model asks: "which other words in this sentence should I pay attention to when understanding this word?"

In the sentence above, when processing "it", the model learns to pay high attention to "animal" and low attention to "street". This is how context within a sentence gets captured.

Step 5 — How your chunks actually get embedded
The embedding model you're using (all-MiniLM-L6-v2) is a transformer that was trained in two phases:

Phase 1 — Pretraining: It read hundreds of millions of sentences and learned language generally — grammar, facts, relationships between concepts.

Phase 2 — Fine-tuning for similarity: It was then specifically trained on pairs of sentences labeled as similar or dissimilar. For each pair, it learned to produce vectors that are close together for similar pairs and far apart for dissimilar ones.

So when you pass a chunk of text to it:

python
embed_model.encode("The patient should avoid consuming sodium")
Internally the model:

Breaks the text into tokens (roughly words or word-pieces)
Converts each token to an initial vector
Runs it through 6 layers of self-attention — each layer refines the vectors by incorporating context from surrounding tokens
After the final layer, pools all the token vectors together into one single vector representing the whole chunk
That final vector — typically 384 numbers for MiniLM — is the embedding. It encodes the meaning of the entire chunk as a point in 384-dimensional space.

Step 6 — Why similar meaning = close vectors
After fine-tuning on millions of similar/dissimilar pairs, the model has learned to organize vector space so that:

"avoid sodium"          → [0.23, -0.54, 0.87, ...]
"don't eat salty food"  → [0.21, -0.51, 0.90, ...]  ← very close
"how to fix a car"      → [-0.67, 0.34, -0.12, ...] ← far away
The angle between the first two vectors is tiny. The angle between either of them and the third is large. This is what cosine similarity measures — and it's why RAG retrieval works across different phrasings of the same idea.

The full picture in one flow
Your PDF chunk:
"The patient should avoid consuming sodium"
          ↓
Tokenizer splits into tokens:
["The", "patient", "should", "avoid", "consuming", "sodium"]
          ↓
Each token gets an initial vector (from a lookup table)
          ↓
6 layers of self-attention run
(each layer: every token looks at every other token,
 refines its own vector based on context)
          ↓
Final token vectors pooled into one vector
          ↓
[0.23, -0.54, 0.87, 0.12, ... ] (384 numbers)
          ↓
Stored in ChromaDB alongside the original text
Same process runs on the user's query. Then cosine similarity finds which stored vectors are closest to the query vector. Those chunks get retrieved.

The key takeaway: you don't program any of this yourself. The embedding model already knows how to do all of it. You just call embed_model.encode("your text") and get back the vector. All the transformer layers, attention mechanisms, and training happened before you ever touched the model.
